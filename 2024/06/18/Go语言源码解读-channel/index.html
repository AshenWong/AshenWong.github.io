Go语言源码解读-channel

发表于2024-06-18|更新于2024-06-18|GOLANG
|阅读量:

首先简单介绍一下golang的channel结构，channel是golang中的一个通信机制，通过它可以让一个goroutine给另一个goroutine发送值信息。通过make(chan int) 就可以创建一个无缓冲的可以发送int值的channel，通过make(chan int, 3) 就可以创建一个缓冲区大小为3的channel。

数据结构

首先来看一下定义channel的结构体

值得注意的是其中buf成员以及recvq, senq成员，buf是一个指向实际存储通道元素的循环队列的指针，recvq, senq两个队列分别存放了当前阻塞的接收goroutine和发送goroutine。

接下来再看一看waitq结构体

waitq是一个双端队列，内部的节点是通过sudog结构体定义的，sudog表示当前的goroutine相关的信息。

通道操作

发送数据

向通道发送数据c <- elem 实际上是通过chansend(c, elem, true, getcallerpc()) 函数实现的，接下来看看chansend()函数的实现细节以及向通道发送数据的几种情况：

  1. 向nil通道发送：当前goroutine会被永久阻塞，并报错fatal error: all goroutines are asleep - deadlock!

  2. 向已关闭的通道发送：引发panic

  3. 有接收gorountine阻塞时：调用send函数，将数据直接发送给该goroutine

  4. 向带缓冲区且缓冲区未满的通道发送：将数据存放在缓冲区中

    通过比较 qcount 和 dataqsiz，确定 hchan.buf 是否有可用空间。将 ep 指针指向的区域复制到要发送的循环队列缓冲区中，并调整 sendx 和 qcount，以等待元素的发送。

  5. 缓冲区已满：阻塞

    在当前栈上创建一个goroutine对象，调用acquireSudog 将当前goroutine置为阻塞状态，并将其添加到sendq队列中，gopark()函数将当前goroutine阻塞。

channel发送操作总结：

  1. 发送时锁定整个channel结构
  2. 从recvq中获取（dequeue）等待接收的goroutine，并将数据发送给该goroutine
  3. 若recvq为空，且缓冲区不满，将数据暂存至缓冲区
  4. 若recvq为空，且缓冲区已满或无缓冲（无缓冲通道实际就是缓冲区大小为0的缓冲通道，实现细节没有任何差别），则阻塞。
  5. send goroutine阻塞时，写入的内容会保存在当前goroutine的sudog结构中，并且当前的 goroutine 会入队 sendq。

接收数据

channel接收与发送操作的情况类似，但有两点值得注意：

  1. 当channel被关闭后，无法发送数据，但仍然能够从中接收数据。
  2. 若果channel关闭后无剩余数据，但仍有goroutine从中接收数据，会接收到当前通道数据类型的零值。

通道关闭

  1. 关闭空的通道：panic

  2. 关闭已关闭的通道：panic

  3. 正常情况：所有该通道上的接收goroutine和发送goroutine进入就绪状态，等待调度，并且发送goroutine会panic。

select

接下来看看select操作

select语句中的每个case都是由一个scase结构体表示，并汇总成一个scases数组。

  1. select中的操作是互斥的，所以需要获取select语句涉及到的所有通道的锁，具体做法是根据hchan结构体的地址排序获取锁，所以不会一次性给所有通道上锁。

  2. 对所有涉及的通道进行洗牌，以提供伪随机的轮询顺序保证，并根据轮询顺序依次逐一遍历所有case，查看其中是否有任何通道已准备好进行通信。这种轮询顺序使得select操作不一定遵循程序中声明的顺序。

  3. 只要有通道操作不阻塞，select语句就可以返回，并且未必需要轮询所有通道

  4. 如果当前没有通道响应，也没有default语句，则当前goroutine阻塞并且根据其情况挂载在所有通道的相应等待队列上。

  5. select操作期间的接收、发送和关闭操作与通道上的接收、发送和关闭的一般操作类似。
